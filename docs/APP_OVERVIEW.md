## 1. Пользовательский путь

### Регистрация → логин

1. На экране Login/Register пользователь вводит login и password.
2. Регистрация: `POST /register` — создаётся пользователь с хэшем пароля.
3. Вход: `POST /login` — в ответ приходят `access_token` и `refresh_token` (JWT). Токены хранятся **только в памяти** вкладки (не в `localStorage`).
4. При `401` на API клиент один раз пробует `POST /refresh`; при повторном отказе сессия сбрасывается.

### Создание / поиск чата

1. В сайдбаре — кнопка создания чата.
2. **Личный:** поиск пользователя (`GET /users/search?login=`), затем `POST /chats` с `type=direct`. Если чат уже есть — сервер вернёт существующий.
3. **Группа:** название + `POST /chats` с `type=group`; создатель становится `admin`.
4. Список чатов — `GET /chats` (превью последнего сообщения, курсор своего прочтения, сортировка по активности). Клиентский фильтр в сайдбаре по уже загруженному списку.

### Переписка в реальном времени

1. Открытый чат: лента сообщений, поле ввода (Enter — отправить, Shift+Enter — новая строка).
2. При отправке клиент генерирует `client_msg_id` (UUID), показывает оптимистичное сообщение со статусом «ожидает ack» (◌).
3. Сообщение уходит по WebSocket; после `ack` статус становится «доставлено на сервер» (✓); когда все остальные участники отметили прочтение — ✓✓.
4. Входящие от других — кадр `new_message` (если чат открыт — в ленту; иначе обновляется превью в сайдбаре; неизвестный `chat_id` — тихий refetch списка чатов).

### История / пагинация

1. При открытии чата — первая страница истории (`GET …/messages`, без `before_id`).
2. Скролл к верху ленты — следующая страница с `before_id` = id самого старого загруженного сообщения.
3. Поиск по чату (`SearchPanel`) — `GET …/search?q=`; клик по результату скроллит к сообщению, при необходимости догружая историю до нужного id.

### Управление участниками группы

1. Панель участников (slide-in): список с ролями, online/last_seen.
2. Только admin видит добавление (поиск логина) и удаление участников.
3. Admin может переименовать группу — у остальных приходит WS `chat_updated`.

### Редактирование сообщений и реакции

1. Свои сообщения можно отредактировать (inline); на сервере выставляется `edited_at`, участникам — WS `message_edited`.
2. Реакции 👍 / 👎 / ❤️: повторный клик той же снимает реакцию; смена типа заменяет. Синхронизация счётчиков — WS `reaction_updated`.

### Профиль (данные и тема)

1. Экран Profile: просмотр `GET /me`, смена login (`PATCH /me`), смена пароля (`PATCH /me/password` с проверкой текущего).
2. Переключатель светлой/тёмной темы (предпочтение в `localStorage`; токены туда не пишутся).
3. После смены login онлайн-пользователи с общими чатами получают `user_updated`.

## 2. Что происходит технически на каждом шаге

| Шаг пользователя | Технически (кратко) |
|------------------|---------------------|
| Регистрация | Service валидирует login/password → Argon2id hash → `INSERT users`. |
| Логин | Verify Argon2id → выпуск пары JWT (короткий access, длинный refresh, разные секреты). |
| Защищённый REST | Middleware парсит Bearer access JWT → `user_id` в контекст → service. |
| Создание direct | Транзакция: insert `chats` + два `chat_members`; при unique conflict — `GetDirectByUsers`. |
| Создание group | Транзакция: insert group + создатель с `role=admin`. |
| Отправка сообщения | Клиент: UUID `client_msg_id` + очередь WS. Сервер: проверка членства → `INSERT … ON CONFLICT` → **синхронный ack** в то же соединение → broadcast `new_message` онлайн-участникам (кроме отправителя). |
| История | Service проверяет членство → `ListByChat` + батч `GetReactionSummaries`. |
| Mark read | Upsert `chat_read_state` с `GREATEST` → WS `read` остальным онлайн-участникам. |
| Add/remove member | Проверка `type=group` + `role=admin` → insert/delete `chat_members`. |
| Поиск в чате | Членство → `ILIKE` по `body`, limit 50. |
| Edit / reaction | REST → service → БД → `RealtimeNotifier` (WS hub) без импорта hub из service. |
| Presence | Первое WS-соединение пользователя → `presence online`; последнее закрытие → запись `last_seen_at` → `presence offline`. |

## 3. Real-time слой в целом

### Hub

In-memory структура: `user_id → множество WS-соединений` (несколько вкладок). Регистрация/снятие соединения; `IsOnline` для REST members; broadcast по списку `user_id` из `chat_members` (или shared-chat peers для `user_updated` / `presence`).

### Аутентификация

1. HTTP upgrade на `GET /ws`.
2. Первым текстовым фреймом клиент шлёт `{"token":"<access_token>"}`.
3. Сервер ждёт не дольше **5 секунд**; иначе close (policy violation).
4. Невалидный/пустой токен — соединение закрывается.
5. После успеха — register в hub, presence online при первом соединении, цикл чтения кадров (`send_message`).

### Разрыв и реконнект (клиент)

1. При обрыве статус UI → `reconnecting`, exponential backoff (от ~1 с до ~30 с).
2. Неотправленные исходящие сообщения остаются в локальной очереди с теми же `client_msg_id`.
3. После `onopen` снова шлётся auth-фрейм с актуальным access-токеном, затем flush очереди.
4. Повторная доставка на сервер безопасна за счёт `UNIQUE (chat_id, client_msg_id)` и ack с тем же `server_id`.

### Серверный shutdown

HTTP `Shutdown` + `hub.Shutdown` — закрытие активных WS при остановке процесса.

## 4. Идемпотентность и оффлайн-получатели

### Идемпотентность отправки

- Клиент **обязан** сам генерировать `client_msg_id` до первой попытки и не менять его при retry/реконнекте.
- БД: уникальный ключ `(chat_id, client_msg_id)`.
- Повторный `INSERT` с тем же ключом возвращает существующую запись (без второго сообщения).
- Ack всегда содержит серверный `id` сохранённой строки — клиент переводит ◌ → ✓.

### Онлайн-получатели

После успешной записи сервер рассылает `new_message` всем **сейчас подключённым** участникам чата (кроме отправителя). Это best-effort push в память процесса; задержка целевая для MVP — низкая (без брокера).

### Оффлайн-получатели

- Persistent-очереди / push-уведомлений **нет** (осознанно, см. [PROJECT_SUMMARY.md](./PROJECT_SUMMARY.md) §5).
- Пропущенные сообщения остаются в PostgreSQL.
- Получатель видит их через **REST-историю** при открытии чата / пагинации (и через последующие live-`new_message`, если уже онлайн).
- Явной клиентской «догрузки разрыва» при реконнекте, пока чат уже открыт, в коде **не найдено** — поведение в этом краевом случае **требует проверки** вручную.

### Прочтение

- Курсор в БД — источник истины; «прочитано всеми» на UI выводится сравнением курсоров участников с `message.id`, без отдельного серверного флага на сообщении.

## Краткая схема потока сообщения

```
Клиент                    Сервер (WS)              PostgreSQL              Другие онлайн-клиенты
  |                          |                         |                          |
  |-- send_message --------->|                         |                          |
  |   (client_msg_id)        |-- INSERT ON CONFLICT -->|                          |
  |                          |<------- row ------------|                          |
  |<------- ack -------------|                         |                          |
  |                          |-- new_message ------------------------------------->|
```
