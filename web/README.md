# Messenger — веб-клиент

React SPA для мессенджера. Стили — CSS-модули, дизайн-токены в `src/styles/tokens.css`.

## Запуск

```bash
# из корня репозитория — поднять Postgres и API
docker compose up -d
cp .env.example .env
# применить миграции (нужен golang-migrate)
migrate -path migrations -database "$DATABASE_URL" up
go run ./cmd/server

# фронтенд
cd web
npm install
npm run dev
```

Откройте http://localhost:5173. В dev-режиме Vite проксирует REST и WebSocket на `http://localhost:8080` (см. `vite.config.ts`).

Сборка:

```bash
npm run build
```

Для production задайте базовый URL API через переменную окружения:

```bash
VITE_API_URL=https://api.example.com npm run build
```

Проверка API против живого бэкенда (вспомогательный скрипт):

```bash
python3 web/scripts/dod_api_check.py
```

## Хранение токенов

Access- и refresh-токены хранятся **только в оперативной памяти** (модуль `src/api/auth.ts`, состояние синхронизируется с React через `AuthContext`).

**Почему не localStorage:**

- При XSS злоумышленник может прочитать persistent storage через JavaScript; токен в памяти хотя бы не переживает перезагрузку вкладки и не лежит в общем хранилище браузера.
- Бэкенд не поддерживает server-side revoke refresh-токенов — сессия и так ограничена TTL JWT; дополнительная персистентность на клиенте не даёт существенного выигрыша в безопасности.

При закрытии вкладки потребуется повторный вход. При ответе API `401` клиент один раз пробует `POST /refresh`; при повторном `401` сессия сбрасывается и показывается экран входа.

## Responsive (≤768px)

На узком экране сайдбар скрыт по умолчанию и открывается как **оверлей** поверх чата (кнопка `☰` в шапке окна чата). Клик по затемнённому фону или выбор чата закрывает сайдбар. Панель участников и поиск по чату остаются slide-in / inline в основной области — горизонтальный скролл не появляется.

## Реализованные экраны

1. **Login / Register** — валидация, ошибки API (409, 401)
2. **Layout** — сайдбар + чат, индикатор WS (`● online` / `○ reconnecting`)
3. **Список чатов** — клиентский поиск, превью и время последнего сообщения
4. **Окно чата** — лента с подгрузкой вверх (`before_id`), пузыри свой/чужой
5. **Поле ввода** — textarea с автовысотой, Enter / Shift+Enter, `client_msg_id` (uuid)
6. **Панель участников** — slide-in справа, add/remove для admin (поиск по логину)
7. **Поиск по чату** — `GET /chats/{id}/search`, скролл к результату с дозагрузкой истории
8. **Создание чата** — кнопка «+» в сайдбаре, модалка: личный (поиск пользователя) / группа; после `POST /chats` чат сразу попадает в локальный список сайдбара

WebSocket: аутентификация первым фреймом, обработка `ack` (◌→◐) и `new_message`, реконнект с backoff и локальная очередь неотправленных сообщений.

### Синхронизация списка чатов

| Событие | Поведение клиента |
|---|---|
| Успешный `POST /chats` | Ответ сразу в начало локального списка (`upsertCreatedChat`), без ожидания WS/рефетча |
| `new_message` по уже известному `chat_id` | Обновление превью и сортировка сайдбара |
| `new_message` по неизвестному `chat_id` | Тихий `GET /chats` и подстановка чата в список (новый direct / первое сообщение после добавления в группу) |

## Известные ограничения (вне скоупа)

Согласно ТЗ (`docs/spec.md` §9 и `frontend_prompt_messenger.md` §4), **не реализовано**:

- read receipts (кроме ◌/◐ ack от сервера)
- push-уведомления
- вложения / медиа
- редактирование и удаление сообщений
- светлая тема
- нативные мобильные жесты (swipe-back и т.п.)
- **WS-событие о добавлении в группу** — в spec только `ack` / `new_message`. Если вас добавили через `POST /chats/{id}/members` и никто ещё не писал, чат не появится в сайдбаре до первого `new_message`, ручного перехода или релогина. Чинится отдельным этапом на бэкенде (не в рамках клиентского фикса списка чатов).
